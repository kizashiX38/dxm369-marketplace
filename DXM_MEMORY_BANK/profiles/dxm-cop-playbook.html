DXM-C.O.P. Playbook

Split the brain. Externalize the memory. Compress the context. This playbook defines how GPT and Claude operate as a single distributed cognition engine under the Operator's control.

Split Brain System
Architecture

GPT handles strategy, reasoning, and compression. Claude focuses on execution and minimal diffs. The Memory Bank holds all long-term state so no model carries history.

[Operator] ↓ goals [ GPT Brain (Think) ] ↓ atomic tasks [ Claude Brain (Do) ] ↓ changes [ World + Memory Bank ] ↑ feedback to GPT
Principle: Never send Claude the whole story. Send Claude only the line it needs to rewrite.

01 · Roles & Responsibilities

Operator (You)

You define the mission, choose tools, and keep the loop tight.

  • State the outcome, not the feeling.
  • Decide what belongs in Memory Bank vs ephemeral chat.
  • Enforce atomic tasks for Claude.

GPT Brain (Strategy Layer)

Used for thinking, not typing.

  • Analyze problems and design step-by-step plans.
  • Summarize and compress large context into Claude-sized shards.
  • Generate exact prompts and patch instructions for Claude.

Claude Brain (Execution Layer)

Used for diffs, not essays.

  • Apply patches to specific files or blocks only.
  • Return minimal output: diffs, code blocks, short confirmations.
  • Avoid high-level reasoning and architecture decisions.

02 · Memory Bank Schema

The DXM Memory Bank is the external cortex. It holds everything the models should not carry: specs, rituals, baselines, session logs, and decisions.

Rule: If it may be needed again in 3 days, it belongs in the Memory Bank, not in the scrollback.

03 · Operational Flow

Step 1 · Define Outcome

Start every session by writing a one-line outcome and optional constraints.

Outcome: "Make admin /bridge-fetch route safe and idempotent." Constraints: - "No new features." - "Use existing apiSafe() helper." - "No changes to logging yet."

Step 2 · Think with GPT

Send code/context to GPT and ask for a surgical plan and Claude-ready prompt.

➤ Identify exact files and lines to touch.
➤ Produce a compressed view: only the relevant functions or blocks.
➤ Generate an atomic instruction for Claude.

Step 3 · Execute with Claude

Use a preset header to keep Claude in execution mode.

DXM-C.O.P. ACTIVE You are the execution engine only. No reasoning or architecture. Your job: precise diffs on the files I give you. If a file is not shown, ignore it. Return only the patch or updated code. NEXT ACTION → "Wrap the GET handler in src/app/api/admin/bridge-fetch/route.ts with apiSafe(handler). No other changes."

Step 4 · Log the Outcome

After a meaningful change, capture it in a session log inside the Memory Bank.

sessions/2025-12-08_claude-dev_bridge-fetch-hardening.md --- project_id: dxm369-marketplace llm: claude-code files_touched: - src/app/api/admin/bridge-fetch/route.ts decisions: - "Standardized error response via apiSafe." --- • Applied wrapper to GET handler. • Verified response shape against api-contracts.md.

04 · Runbook Checklists

Before Calling Claude

☑ Is the task atomic (one file, one handler, one change)?
☑ Did GPT already do the architecture thinking?
☑ Do you have a minimal code slice ready, not the entire repo?

After Claude Responds

☑ Apply the diff and run tests / basic checks.
☑ Capture important decisions into the Memory Bank.
☑ Close the loop with GPT if next steps are needed.

Anti-Patterns

  • "Fix everything in this project" → guaranteed context bloat.
  • Feeding Claude full repos repeatedly instead of diff slices.
  • Letting important knowledge live only in chat scrollback.
Guardrail: If the prompt feels like a paragraph, it probably needs to be split into two tasks.
